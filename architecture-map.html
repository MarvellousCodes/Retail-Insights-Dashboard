🧠 Core Components

App.tsx — Central State Hub
The god component. Owns ALL application state and passes everything down as props.
INPUT File uploads from UploadPage, manual product entries, threshold edits, source toggles
OUTPUT Props to every page: products[], marginAlerts[], priceAnomalies[], thresholds[], fixedIds, sourceControls
STATE sources[], activeSourceIds, marginAlerts[], priceAnomalies[], thresholds[], productOverrides, fixedIds, tab, theme
⚠️ Problems

• God component anti-pattern — 250+ lines managing 10+ state variables. Every state change re-renders the entire app tree.
• No persistence — all data lost on refresh. No localStorage, no IndexedDB, no backend save.
• No URL routing — tab state is internal useState, can't bookmark or share links.
• Types defined inline — Product, MarginAlert, Source types live in App.tsx instead of a shared types file.
• Analysis runs on every render — useEffect re-runs runAnalysis on any activeProducts/thresholds change with no debounce.

runAnalysis() — Analysis Engine
Pure function that computes margin alerts and price anomalies from products + thresholds.
INPUT products: Product[], thresholds: DeptThreshold[], productOverrides: Record<string, number>
OUTPUT { marginAlerts: MarginAlert[], priceAnomalies: PriceAnomaly[] }
⚠️ Problems

• O(n×m) threshold lookup — for each product, scans entire thresholds array with .find(). Fine for 4K SKUs, breaks at 100K+.
• Hardcoded severity bands — gap≥10 = High, gap≥5 = Medium. Not configurable per department.
• No historical comparison — can't detect margin drift over time, only snapshot analysis.
• roundRetailPrice() is Irish-specific — .49/.99 rounding won't work for other markets.

📄 Pages

DashboardPage — Overview Dashboard
KPI cards, department breakdown bars, severity distribution, sparkline charts. The landing page.
INPUT products[], marginAlerts[], priceAnomalies[], thresholds[], fixedIds, sourceControls
OUTPUT Visual-only (no callbacks except onNewUpload)
⚠️ Problems

• 18KB single file — inline Sparkline component, all chart logic, all layout in one file.
• No real-time data — static snapshot, no WebSocket or polling for live updates.
• Sparkline data is synthetic — generates random-ish values, not actual historical trends.

UploadPage — CSV Upload & Parse
Drag-and-drop CSV upload with auto column mapping, multi-file support, manual product entry form.
INPUT CSV files (drag/drop or file picker), manual product form data
OUTPUT onAnalyse(files[]), onManualAdd(product), onRemoveSource(id)
⚠️ Problems

• 30KB monster file — CSV parser, column mapper, preview table, manual entry form all in one.
• Client-side CSV parsing — custom parser handles delimiters/quotes but no streaming for large files.
• Column mapping is heuristic — fuzzy matches on header names, can silently mismap columns.
• No validation feedback — bad rows silently produce 0-value products.

IssuesPage — Issue Tracker
Filterable, sortable table of all margin alerts and price anomalies. Mark-as-fixed, per-product threshold overrides.
INPUT products[], marginAlerts[], priceAnomalies[], fixedIds, productOverrides, sourceControls
OUTPUT onMarkFixed(id), onSetProductOverride(id, margin)
⚠️ Problems

• 27KB file — filtering, sorting, inline editing, severity badges all in one component.
• No pagination — renders all issues in a single scrollable list. Sluggish at 1000+ alerts.
• Fixed state is ephemeral — marking an issue "fixed" is lost on refresh.

ReportsPage — Export Reports
Generates attention reports with top-N worst margins, anomalies, department summaries. CSV download.
INPUT products[], marginAlerts[], priceAnomalies[], fixedIds, sourceControls
OUTPUT CSV file download (browser-generated)
⚠️ Problems

• No PDF export — only CSV. The Margin Guardian backend has PDF generation but it's not connected.
• No email delivery — Margin Guardian has email but this frontend doesn't use it.
• Report is regenerated every render — no caching or memoization.

InsightsPage — Product Intelligence
Metric cards showing top/bottom performers by margin and profit. Expandable product lists.
INPUT products[]
OUTPUT Visual-only
⚠️ Problems

• No trend analysis — shows current snapshot only, no comparison to previous uploads.
• No supplier analysis — doesn't group by supplier to identify problematic vendors.

DepartmentsPage — Department Breakdown
Per-department health grades (A-F), margin averages, profit totals, below-target counts.
INPUT products[], thresholds[]
OUTPUT Visual-only
⚠️ Problems

• Grade thresholds are hardcoded — A≥40%, B≥30% etc. Not configurable.
• No drill-down — can't click a department to see its products.

InvoiceScannerPage — OCR Invoice Entry
Upload PDF/JPG/PNG invoices, OCR extracts line items, editable table, CSV export for Retail Solutions import.
INPUT Invoice file (PDF/JPG/PNG) → POST to Margin Guardian API at localhost:5000
OUTPUT Editable extraction table, CSV download
⚠️ Problems

• Hardcoded API_BASE — "http://127.0.0.1:5000" is baked in. No env var, no config.
• Disconnected from main analysis — scanned invoices don't feed into the margin analysis pipeline.
• No error recovery — if OCR fails, user gets a generic error with no retry or manual fallback.
• Depends on separate Flask backend — the only page that needs a server, creating a split architecture.

SettingsPage — Threshold Configuration
Edit department minimum margin thresholds. Add/remove departments. Reset to defaults.
INPUT thresholds[]
OUTPUT onUpdate(thresholds[])
⚠️ Problems

• No persistence — custom thresholds lost on refresh.
• No import/export — can't save threshold profiles or share between stores.

🧩 Shared Components

Sidebar — Navigation
Collapsible sidebar with 8 nav items, theme toggle, user avatar. Controls which page is shown.
INPUT activeTab, theme, expanded state
OUTPUT onTabChange(tab), onToggle(), onThemeToggle()
⚠️ Problems

• Hardcoded user — shows "Store Manager / Pro plan" with no auth system.
• No mobile responsive — sidebar doesn't collapse to hamburger on small screens.

SourceSelector — Data Source Picker
Dropdown to toggle which uploaded CSV files are included in the active analysis.
INPUT sources[], activeIds Set
OUTPUT onToggle(id), onSelectAll(), onSelectNone(), onRemove(id)
⚠️ Problems

• No search/filter — with many uploads, no way to find a specific source.
• Click-outside detection — manual mousedown listener instead of using a proper dropdown component.

⚙️ Backend & Infrastructure

API Server (Express) — Unused Backend
Express server with CORS, pino logging, health check endpoint. Scaffolded by Replit but never used.
INPUT HTTP requests on PORT env var
OUTPUT GET /api/healthz → { status: "ok" }
⚠️ Problems

• Completely empty — only a health check route. No product, analysis, or report endpoints.
• Requires PostgreSQL — DB lib throws if DATABASE_URL not set, but nothing uses the DB.
• Dead code — the frontend does everything client-side. This server adds complexity for zero value.

DB Schema (Drizzle/PostgreSQL) — Empty
Drizzle ORM setup with PostgreSQL pool. Schema file is a template with no tables defined.
INPUT DATABASE_URL env var
OUTPUT Nothing — no tables, no queries
⚠️ Problems

• Zero tables defined — schema is commented-out example code.
• Unnecessary dependency — PostgreSQL requirement blocks local dev for no benefit.

Margin Guardian API (Flask :5000) — OCR Backend
Separate Flask app providing invoice OCR via Tesseract + pdfplumber. Only used by InvoiceScannerPage.
INPUT POST /api/invoice/upload with multipart file
OUTPUT JSON array of extracted line items
⚠️ Problems

• Separate process — must run independently on port 5000. No unified dev command.
• No CORS config visible — may fail silently from browser.
• Tesseract dependency — requires system-level install, not portable.

generate_test_data.py — Test Data Generator
Python script generating 20K realistic Irish retail products across 8 departments with realistic margins.
INPUT Hardcoded config (department counts, margin ranges, supplier lists)
OUTPUT test_data_20k.csv
⚠️ Problems

• Hardcoded output path — absolute path to user's machine.
• No CLI args — can't control row count, departments, or output path.

🎯 Priority Problem Areas (Tackle Order)

1. No Data Persistence

Every upload, threshold edit, and "mark fixed" action is lost on page refresh. This is the #1 blocker for real usage. Fix: Add IndexedDB/localStorage persistence for sources, thresholds, overrides, and fixed IDs. Or wire up the empty Express backend with actual CRUD endpoints.

2. God Component (App.tsx)

App.tsx owns 10+ state variables and passes them as props through every page. Fix: Extract into React Context or Zustand store. Split types into shared types file. Add URL routing (wouter is already a dependency but unused).

3. Invoice Scanner is Disconnected

Scanned invoices produce a table but don't feed back into the margin analysis. Two separate worlds. Fix: Add "Import to Analysis" button that converts scanned items to Product[] and adds them as a source.

4. Monster Page Files

UploadPage (30KB), IssuesPage (27KB), DashboardPage (18KB) are unmaintainable monoliths. Fix: Extract CSV parser, column mapper, chart components, filter/sort logic into separate modules.

5. Dead Backend Code

Express server + Drizzle DB + PostgreSQL requirement exist but do nothing. Fix: Either remove entirely (pure SPA) or build real API endpoints for persistence, multi-user, and report generation.

6. No Historical Analysis

Can only analyze current snapshot. No margin drift detection, no week-over-week comparison, no trend alerts. Fix: Store upload history with timestamps, compute deltas between uploads.

7. No Authentication or Multi-Store

Hardcoded "Store Manager / Pro plan" with no login. Can't support multiple stores or users. Fix: Add auth layer if going multi-tenant, or at minimum make store name configurable.

8. Performance at Scale

Client-side analysis works for 4K-20K products but will choke at 100K+. No virtualized lists, no pagination, no Web Workers. Fix: Add virtual scrolling to IssuesPage, move analysis to Web Worker, add pagination.

Security architecture

Three locks between queries and your data

Every query passes through three independent safety layers. If any single lock fails, the query is blocked. Your POS data cannot be modified, exported, or used for model training.

1

SQL Validation

Only SELECT queries pass. INSERT, UPDATE, DELETE, DROP are blocked before they reach the database.

2

Read-Only Database Role

Queries connect through a restricted database user with SELECT permission only. Writes are impossible at the database level.

3

Read-Only Transaction

The connection enforces default_transaction_read_only=on. Even a compromised query cannot write. Belt, braces, and padlock.

Common questions

Answers before you have to ask

Can my data be changed?

No. Three independent locks prevent any write operation. The query user has SELECT-only permissions, queries are validated before execution, and the connection enforces read-only mode.

Do I need new hardware?

No. RetailGuard works with your existing CBE or Pervasive POS. No new till, no barcode printer, no installation on your PC.

Where does my data go?

Nowhere outside the EU. Processing happens in-region. Your data is never used for model training, never exported, and never shared with third parties.

What if usage costs run up?

It cannot. Every feature has hard daily and monthly caps. Total usage costs approximately €1/month. You will never be surprised.

Is it hard to set up?

One afternoon. We handle the connection. You see your first margin report within 24 hours. No changes to your POS, no downtime.

Works with my POS system?

Yes. We support CBE, Pervasive PSQL, and most SQL-based Irish POS systems. Compatible with GALA, BWG, and Musgrave symbol groups.
